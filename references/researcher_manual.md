1. 善于利用已经获得的实验结果, 总结其中的规律, 用以加速未来的实验设计.

例: 在对照组获得的调参经验可以用于指导实验组, 虽然不能直接确定实验组什么超参好, 但已足够用于确定中心点和缩小搜索范围.
例: 在数据集 1 上的经验也可以用于指导数据集 2.

2. 善于利用最新的知识指导实验.

例: 调参时, 能想到:"我这个情况虽然不是 muP, 但可以利用 muP 的思想, 所以这个参数在这种情况下应该更小 / 更大".

3. 善于复用之前的实验.

例: 很多实验设计或 hp 扫描往往与之前的实验有一些重叠, 此时就可以在实验前先做一些分析, 预测想法靠不靠谱.
例: 实验设计时复用之前的结果节约时间.

4. 聚焦最可能方向时, 也不忘那些被忽略的方向, 伟大的发现往往就在被忽略的角落.

例: 调超参时想到 "一般人们都认为是这样这样, 但万一是那样那样呢? 实验又不贵, 我要不试一下?"

5. 小步快走, 实验是一点一点做的, 不断总结修正, 而不是一次做一个大的错的.

例: 先在几个 case/数据集 + 小规模网络/step 上尝试, 再慢慢扩大规模, 而不是一上来就跑一个全尺寸 + 所有数据集, 等12h收集回来发现都是垃圾.
例: <wrong>"一上来就全规模实验, 花了 $2000 OpenAI API 美美送给 reviewer 看, 结果代码有 bug 实验全错"</wrong> <correct>"先做 1/10 规模实验, 花 $200 OpenAI API, 送给 reviewer 发现代码有 bug, 修复后再跑 1/10 规模实验, 给 reviewer 看, reviewer 认可后, 再做全规模实验"</correct>
例: 一个 idea 已经在部分 case 上获得了 positive result 之后, <correct>立即将这个 idea 应用于更多 case, 或者 extend 这个 idea</correct>, <wrong>一定要用更多种子反复测试这几个 work 的 case</wrong>. 你要明白: 你看到一篇文章最后有多个 seed 的大量实验不意味着作者一开始就一直在做多个 seed 的实验. 实验规划要分清主次, 多 seed 的实验, 大规模的实验都是最后再补齐的.

6. 追求完美

例: 之前计划好的某一个 step, 不是走过场做完就完了, 而是做完想: "我这个有没有做到完美", 然后不断打磨, 直到完美再做下一步.

7. 代码 unit test 都过了 ≠ 实验 setup 没 bug

代码能跑不代表实现的就是一开始的 idea. 实验前几次都是 negative 也不代表 idea 不 work — 很可能只是代码实现错了, 还没碰到 idea 本身.
参见 5. 小步快走, 这也就是小步快走的理由: 前几次实验绝大概率都是错的, 是需要慢慢修代码的, 此时全尺寸跑只会浪费时间和钱.

8. 公平比较常见误区

例: 对照组已经和文献结果对上了, 并且还比文献结果稍好一点. <wrong>还浪费算力疯狂调对照组</wrong> <correct>将精力集中在实验组的调优/trick上</correct>
注: LLM 容易陷入 "为公平起见两侧都重跑" 的执念. 但对照组只要对上 baseline 数字就够 fair, 不需要再优化. 把多余算力花在实验组上.

例: 对照组和实验组是两种不同的训练方案/神经网络类型, <wrong>说"为了公平比较它们俩的 lr 要一样"</wrong> <correct>整理清楚思路, 想明白我们的实验是要验证什么, 是为了什么, 就会明白应该是 "各自扫描超参, 然后比较各自的最好结果"</correct>
